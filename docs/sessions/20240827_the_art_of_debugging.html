Defensive programming is the best way to solve many (future) bugs: the future-you will praise you. 1. Add pre-conditions to your functions to avoid unexpected and unpredictable errors in later stages (**fail fast principle**: stop the current operation as soon as any unexpected error occurs) ```r # PSEUDOCODE !! if (date < today) stop("Date is too early"); if (class input != numeric vector) stop("Invalid type of input: only numeric vectors allowed.") ``` 2. Assert statements to validate output at intermediate steps, e.g. output of third party libraries that you use in your project. --- class: left, top # Intermezzo: defensive programming Example: apply defensive programming to the `custom_min()` function here below to avoid unexpected behaviors: ``` # Customized `min()` function for NUMERIC(!) vectors # to avoid warning while calculating min() of empty vectors custom_min <- function(x) { if (length(x)>0) min(x) else NA } > custom_min(c("e", "r", "a")) # passing chars should return an error [1] "a" ``` --- class: left, top # Intermezzo: defensive programming - example The package [assertthat](https://rdocumentation.org/packages/assertthat/versions/0.2.1) is very useful while applying defensive programming in your coding life. What we do is adding assertion(s) to our function. ``` library(assertthat) # Customized `min()` function for NUMERIC(!) vectors # to avoid warning while calculating min() of empty vectors custom_min <- function(x) { # check x argument (assertion) assertthat::assert_that( is.numeric(x), msg = "Invalid type of input: only numeric vectors allowed." ) # function core follows if (length(x)>0) min(x) else NA } # Only numeric input are allowed, error arises otherwise custom_min(c("a", "b")) Error: Invalid type of input: only numeric vectors allowed. ``` --- background-image: url({{ site.baseurl}}/assets/images/background_challenge_3.png) class: left, top # Challenge 3 - Debug & defensive programming Let's move to something more familiar: eels tagging data. Do not __hard__code numbers: use variables instead (number of iterations, parameter values in simulations, ...). Easier to read and reduce the amount of potential bugs when you use the same number multiple times. 1. Avoid using **global variables** in your function ```r txt <- "INBO" b <- function(adjective) { paste(txt, "is", adjective) } > my_function("fantastic") [1] "INBO is fantastic" ``` 2. Avoid reusing **global variables** names in your function: very confusing and prone to bugs. ```r txt <- "INBO" my_function <- function() { txt <- "fantastic" paste("coding together is", txt) } > my_function() [1] "coding together is fantastic" > txt [1] "INBO" ``` --- class: left, top # Tips: optimize your code We are typically lazy, aren't we?
