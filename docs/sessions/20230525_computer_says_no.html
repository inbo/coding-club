Debugging starts already while writing code! Defensive programming is the best way to solve bunch of (future) bugs: the future-you will praise you. 1. Add pre-conditions to your functions to avoid unexpected and unpredictable errors in later stages (**fail fast principle**: stop the current operation as soon as any unexpected error occurs) ```r # PSEUDOCODE !! if (date < today) stop("Date is too early"); if (class input != numeric vector) stop("Invalid type of input: only numeric vectors allowed.") ``` 2. Assert statements to validate output at intermediate steps, e.g. output of 3rd party libraries that you use in your project. --- class: left, top # Intermezzo: defensive programming - example Apply defensive programming to the `custom_min()` function here below to avoid unexpected behaviors: ``` # customized `min()` function for NUMERIC(!) vectors to avoid warning while calculating min() of empty # vectors custom_min <- function(x) { if (length(x)>0) min(x) else NA } > custom_min(c("e", "r", "a")) # passing chars should return an error [1] "a" ``` --- class: left, top # Intermezzo: defensive programming - example The package [assertthat](https://rdocumentation.org/packages/assertthat/versions/0.2.1) is very useful while applying defensive programming in your coding life. What we do is adding assertion(s) to our function. ``` library(assertthat) # customized `min()` function for NUMERIC(!) vectors to avoid warning while calculating min() of empty # vectors custom_min <- function(x) { # check x argument (assertion) assertthat::assert_that( is.numeric(x), msg = "Invalid type of input: only numeric vectors allowed." ) # function core follows if (length(x)>0) min(x) else NA } # only numeric input are allowed, error arises otherwise custom_min(c("a", "b")) Error: Invalid type of input: only numeric vectors allowed. ``` --- background-image: url({{ site.baseurl}}/assets/images/background_challenge_3.png) class: left, top # Challenge 3 - Debug & defensive programming Let's move to something more familiar: eels tagging data. Build up code in small pieces, testing along the way, step by step 1. Make big changes in small steps 1. Smaller functions are better: easier to debug, easier to understand --- class: left, top # Miscellaneous tips: do not code _hard_ 1. Do not __hard__code numbers: use variables instead (number of iterations, parameter values in simulations, ...). Easier to read and reduce bugs when you use the same number multiple times 1. Avoid using **global variables** in your function ```r txt <- "INBO" my_function <- function() { paste(txt, "is fantastic!") } > my_function() [1] "INBO is fantastic!" ``` 1. Avoid reusing **global variables** names in your function: vey confusing and prone to bugs. ```r txt <- "INBO" my_function <- function() { txt = "fantastic" paste("coding together is", txt) } > my_function() [1] "coding together is fantastic" > txt [1] "INBO" ``` --- class: left, top # Miscellaneous tips: optimize your code We are typically lazy, aren't we?
